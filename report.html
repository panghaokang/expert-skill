背景与动机

企业专家的隐性知识，如何沉淀？

每个团队都有 go-to person——他们拥有大量隐性知识，但人走了知识就消失了。

😨

知识断层危机

资深专家离职后，其诊断直觉、设计启发、判断框架随之流失。新人遇到同类问题只能从零摸索，团队效率断崖式下降。

💬

隐性知识难表达

专家的判断框架往往是"肌肉记忆"——他们知道该查什么，但很少写成文档。传统 Wiki 无法捕捉这种启发式思维。

💺

LLM 时代的新解法

大语言模型具备强大的模式归纳和推理能力，可以从专家的原始工作对话中提取判断逻辑，编码为可被 AI 调用的结构化知识。

核心命题：构建一个工程化系统，将专家隐性知识结构化、可复用化、可版本化，形成真正可被 AI 调用的企业知识资产。
解决方案

Expert Skill — 知识蒸馏引擎

一个系统化的知识蒸馏框架，从专家的工作记录中提取专业方法论，生成可被 LLM 直接调用的 Skill 文件。

🎯

蒸馏什么？

判断框架（遇到问题怎么想）、决策模式（优先考虑什么、红线在哪）、经验规则（踩过的坑、验证过的结论）、知识结构（按类型组织的方法论体系）。

🛠

不蒸馏什么？

性格脾气（MBTI、星座）、沟通风格（语气、话术）、人际行为（协作偏好）。产出的 Skill 会说"基于专家的方法论，你应该先检查 Y"，而非模仿某个人的语气说话。

分析架构

双通道交叉分析 + 专长专属模板

通用分析器提取基础方法论，专长分析器补充类型特定的知识结构，两份结果合并生成完整知识图谱。

通用分析维度

  • 专业领域 — 负责的系统、技术栈、职责边界
  • 判断框架 — 优先检查项、分析路径、经验规则
  • 决策模式 — 考量排序、技术红线、何时说不
  • 知识结构 — 按专长类型组织的结构化方法论
  • 经验知识库 — 原话引用 + 结论提炼

专长专项维度（以诊断专家为例）

  • 故障模式库 — 经常处理的故障类型及特征
  • 诊断决策链 — 从症状到根因的排查路径
  • 嗅觉检查项 — "一上来就查"的固定检查
  • 工具与命令 — 诊断工具箱及使用场景
  • 常见误判 — 看起来像 A 实际是 B 的模式
关键原则：有依据才写，没有就标注"原材料不足"。有原文依据的结论加引号标注原话。不做性格推断，不分析沟通风格。输出要求具体可执行——不要"可能""倾向于"，要"先查 X，再查 Y，如果 XYZ 同时出现则是 Z"。
产出结构

6 种标准化产物，覆盖完整信息密度谱

每个蒸馏出的专家 Skill 生成从人类可读到机器可解析的完整文件体系，兼顾可读性和可编程性。

skills/expert/{slug}/ ├── SKILL.md ← 可直接被 Claude Code 加载 ├── expertise.md ← 纯专业知识（无身份包装） ├── knowledge_graph.md ← 结构化知识图谱 ├── heuristics.json ← 启发式规则（可编程调用） ├── meta.json ← 生命周期 & 来源元数据 ├── manifest.json ← 安装/分发清单 └── versions/ ← 历史版本归档（≤10）

📖 SKILL.md（入口文件）

完整 Skill 文件，含 frontmatter、专家身份、专业知识、执行规则。用户可直接通过 /expert-skill 或作为普通 Skill 调用。

🔧 heuristics.json（机器接口）

结构化启发式规则，包含 knowledge_format（决策树/设计原则/检查清单/决策框架/Runbook）和 execution_model，可供下游程序化消费。

🗃 manifest.json（分发清单）

安装清单，声明兼容运行时（Claude Code / OpenClaw / Hermes / Codex）、入口点、能力标签和引擎信息。

🔄 meta.json（可审计元数据）

完整元数据——生命周期（created/updated/version/status）、分类（类型/知识格式/执行模型）、来源追溯（knowledge_sources）。

版本管理

持续进化 & 安全回滚

专家知识不是一成不变的。系统支持增量追加新材料，每次更新自动归档，最多保留 10 个历史版本，任意版本一键回滚。

📤 进化模式：追加材料

1

提供新素材

用户提供新文件/消息记录/文档

2

读取现有知识

加载 skills/expert/{slug}/expertise.md

3

自动归档

version_manager backup 保存当前版本

4

增量合并

分析增量内容，合并到专业知识中

5

重建产物

写入 expertise.md + 重建 SKILL.md，v1 → v2

🔃 回滚机制

  • 每次更新前自动备份当前全部产物
  • 最多保留 10 个历史版本，超出自动清理最旧版本
  • 回滚时先备份当前状态，再恢复目标版本
  • 每个版本记录 _info.json（时间戳 + 产物清单 + 备份时间）
  • CLI 一键操作，支持交互式版本选择
$ python3 tools/version_manager.py \ --action rollback \ --slug zhang-san \ --version v3 Backed up zhang-san as version v5 Rolled back zhang-san to version v3
提示词体系

15 个提示词模板的分层架构

每个专长类型拥有独立的 intake → analyzer → builder 三件套，加上通用模板，构成完整的知识蒸馏提示词矩阵。

通用模板（3 个）

intake.md 基础信息录入——4 个引导性问题（称呼/信息/类型/深度描述），含确认汇总流程
analyzer.md 通用专业知识分析——5 个提取维度（领域/判断框架/决策模式/知识结构/经验库），含输出格式模板
builder.md 专业知识构建——将分析结果格式化为标准化 expertise.md，含 5 条生成注意事项

专长模板（5 类 × 3 = 15 个）

troubleshooter/ intake / analyzer / builder
architect/ intake / analyzer / builder
reviewer/ intake / analyzer / builder
decision_maker/ intake / analyzer / builder
operator/ intake / analyzer / builder
应用场景

从个人工具到团队知识库

Expert Skill 在多个层面发挥作用——从 Code Review 到应急响应，从新人培训到架构评审。

🔍

Code Review 增强

加载审核专家的 Skill，自动按检查清单审查代码，识别常见缺陷模式，给出风险等级判定（blocker / warning / nit）。

🛠

故障诊断辅助

加载诊断专家的 Skill，按诊断决策链排查，执行嗅觉检查项，参考故障模式库快速定位根因。

🗃

架构方案评审

加载架构专家的 Skill，用设计原则和技术选型矩阵评估方案，识别反模式和红线违规，生成 ADR。

⚖

技术决策支持

加载决策专家的 Skill，应用决策框架和风险评估矩阵，在不确定条件下做出结构化权衡。

🚨

应急响应 SOP

加载运维专家的 Skill，按标准操作流程处理事故，自动判断升级条件，参考历史事故处理经验。

📚

新人 Onboarding

将多位专家的 Skill 组合成团队知识库，新人即时获取专家的判断框架和方法论，缩短上手周期。
