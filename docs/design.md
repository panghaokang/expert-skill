# 隐性知识挖掘引擎 — 设计规格

日期: 2026-05-11
状态: draft
产品定位: 企业内部知识管理工具 + AI-native 知识蒸馏方法论框架

## 核心概念

### 隐性知识
专家在决策中实际使用但自身未明确意识到的知识，包括：
- **隐性变量**：专家做判断时依赖但从未命名的判断维度
- **隐性优先级**：隐性规则冲突时专家的实际选择次序（口头声称的优先级 ≠ 实际优先级）

### 核心方法论：交叉验证问题三联体

三个机制融合为一个统一的问题设计框架：

- **多角度验证**：锚定问题，建立专家显性规则基准线
- **最小对比对**：表面 70% 重叠但 1-2 个隐性变量不同的场景对，探测专家是否"察觉"隐性变量
- **矛盾暴露**：叠加冲突变量，逼出隐性优先级拓扑

单个三联体的三层结构：

| 层次 | 目的 | 专家反应模式揭示什么 |
|------|------|---------------------|
| A 表面锚点 | 建立基准线 | 显性规则（专家能讲清楚） |
| B 对比探针 | 探测隐性变量辨识度 | 隐性变量是否起作用、专家能否察觉 |
| C 张力引爆 | 逼出优先级拓扑 | 规则冲突时哪个压倒哪个 |

## 系统架构：7 阶段流水线

```
Phase 1 — 基础录入（现有，微调）
Phase 2 — 前置研究 ★ 新
Phase 3 — 隐性变量候选构建 ★ 新
Phase 4 — 三联体问题生成 ★ 新
Phase 5 — 实时访谈 ★ 新
Phase 6 — 访谈分析 ★ 新
Phase 7 — 融合与生成（现有，增强）
```

阶段间依赖：P2 质量 → P3 命中率 → P4 探测有效性 → P6 发现质量。系统瓶颈在 P2/P3。

## Phase 1: 基础录入

复用现有 `prompts/intake.md` 流程：专家名称、专长类型（troubleshooter/architect/reviewer/decision_maker/operator）、领域描述。

## Phase 2: 前置研究

### 输入源
1. **用户主动提供**：简历、论文、演讲资料（Read 工具 + 网络搜索）
2. **原材料导入**：飞书/钉钉消息、群聊、邮件（复用现有采集工具）
3. **开放搜索**：行业背景、技术路线争议、领域内基准案例

### 输出：《先验知识画像》

```yaml
expert_profile:
  identity: { name, title, domain, years_in_field }
  visible_knowledge:   # 材料中可见的专家陈述
    - rule: "..."
  known_decisions:     # 已知决策案例
    - case: "..." context: "..."
  domain_context:      # 领域背景
    key_challenges: []
    common_pitfalls: []
    methodology_clashes: []
  suspected_gaps:      # 材料中未提及但应出现的话题
    - gap: "..."
```

关键设计：`suspected_gaps` 是 P3 的种子——不只看专家说了什么，更看专家没说什么、回避了什么、规则在什么边界外会失效。

## Phase 3: 隐性变量候选构建

### 候选的 4 种来源

| 来源 | 推导方式 |
|------|---------|
| 对比缺失 | 材料中有 A/B 案例但专家未解释判断差异的根因 |
| 领域分歧 | 专家的方法论选择隐入某一阵营但未解释原因 |
| 沉默话题 | suspected_gaps ——同行讨论但专家从未提及 |
| 规则边界模糊 | 专家陈述了规则但未说明边界条件 |

### 输出：《隐性变量候选池》

每个候选包含：
- `id`、`label`、`source_type`
- `evidence_from_profile`：先验画像中的具体证据引用
- `hypothesized_variable`：系统假设的隐性变量（名称、描述、为什么可能是隐性）
- `testability`：是否适合三联体探测

质量门：候选数 5-12 个，每个必须有具体证据支撑，必须有 why_latent 论证。

## Phase 4: 三联体问题生成

### 生成约束

| 约束 | 要求 |
|------|------|
| 表面重叠率 | B 与 A ≥ 70% 文本重叠 |
| 单变量控制 | B 相对 A 只改变 1 个隐性变量；C 在 B 基础上只叠加 1 个冲突变量 |
| 生态效度 | 场景源于专家真实工作领域 |
| 不可预测性 | 专家无法从措辞中猜出被测试的变量 |
| 决策差异应有性 | A 和 B 的理论正确答案应不同 |

### 问题设计模板

```yaml
triplet_group:
  target_variable: lv_001
  domain_context: "..."
  question_A: { text, probes, expected_reveals }
  question_B: { text, variable_changed, probes, expected_reveals }
  question_C: { text, variable_changed, probes, expected_reveals }
```

### 问题组组装策略

开场 1 个（建立信任）→ 核心 3-8 个（高优先级候选）→ 深度 1-2 个（沉默话题来源，可能引发认知冲突）→ 冷却 1 个（安全结束）。

## Phase 5: 实时访谈

### 访谈会话协议

每个三联体组遵循三道提问 + 追问循环：

- **提问 A**：呈现场景 → 记录回答 → 追问 1 轮（思考过程）
- **提问 B**：呈现变体 → 比对 A/B → 追问是否察觉差异及原因
- **提问 C**：呈现冲突 → 追问最大犹豫点和反转条件

### 实时信号检测

| 信号 | 触发条件 | 意义 |
|------|---------|------|
| 察觉信号 | 专家自发说"这个不一样" | 变量处于可察觉但非显性的边界 |
| 迟疑信号 | 长停顿、重复、自我纠正 | 冲突正在发生，隐性优先级被激活 |
| 边界发明 | "要看情况...如果X就A，如果Y就B" | 专家正在现场发明新规则——高浓度隐性知识 |
| 矛盾信号 | C 回答与 A 基准规则直接冲突 | 显性规则可能不是实际使用的规则 |
| 反驳信号 | 专家质疑问题前提 | 系统假设可能错误，同样有价值 |

### 追问规则

```
IF 察觉信号 AND 未说清变量 → 追问"你看出来什么不同让你改判断？"
IF 迟疑信号 → 追问"你在衡量什么？"
IF 边界发明 → 追问"你怎么知道这个边界在哪的？"
IF 矛盾信号 → 直接指出矛盾，请专家解释
IF 模糊词（"经验"、"感觉"、"大概"） → 追问"你感觉的线索是什么？"
```

## Phase 6: 访谈分析

### 单三联体分析

每个三联体产出：
- A 层基准规则提取
- B 层察觉状态判定（`explicit` / `semi_latent` / `deep_latent`）
- C 层优先级拓扑（冲突中哪个规则胜出）
- 提取的隐性知识条目（类型 + 内容 + 置信度）

### 跨三联体分析

| 分析 | 方法 | 产出 |
|------|------|------|
| 隐性变量一致性检验 | 同一变量被 2+ 三联体验证 | confidence medium → high |
| 优先级拓扑一致性 | 同一对冲突规则在不同场景中的胜出关系 | 发现调节变量或标记不一致 |
| 全局边界地图 | 汇总所有"边界发明" | 专家隐性知识的适用域和失效域 |

### 输出：《隐性知识发现报告》

包含：发现的隐性变量（按置信度排序）、隐性优先级拓扑、疑似误判区、未解问题。

## Phase 7: 融合与生成

### 冲突解决规则

| 冲突 | 处理 |
|------|------|
| 访谈陈述 ≠ 材料中的文字 | 以访谈为准 |
| 实际选择 ≠ 声称的选择 | 以实际选择为准 |
| 隐性变量候选被证伪 | 丢弃候选，记录为已验证的无关变量 |

## 关键设计决策

1. **不替代现有流程，而是增强**：现有原材料导入工具变为 P2 的前置研究输入源，现有分析/生成流程在 P7 融合
2. **瓶颈在 P2/P3**：问题质量由前置研究质量决定，系统价值在于能从"已知"推导出"未知"
3. **访谈交互为同步模式**：Claude Code 会话中实时访谈，而非异步问卷
4. **以行为数据为准**：访谈中专家实际做出的选择优于专家口头陈述的规则
