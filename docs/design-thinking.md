# 隐性知识挖掘引擎 — 设计思路

日期: 2026-05-11

---

## 一、问题空间

### 1.1 起点：expert-skill 的当前能力边界

现有的 expert-skill 系统解决了一个明确的问题：**从专家的显性工作痕迹中提取专业知识**。它接收专家的消息记录、文档、邮件，分析其中的判断框架和决策模式，生成可复用的 Skill。

但这个系统有一个根本限制：**它只能提取专家已经表达出来的知识**。

换句话说，它提取的是专家"能说清楚的"和"恰好说过的"，而无法触及专家"说不清楚但实际在用"的知识。

### 1.2 隐性知识为什么重要

企业专家的价值不只在他们的显性知识里。真正让他们稀缺的是：

- 他们能"嗅"到问题的方向，但讲不清闻到了什么
- 他们对两个表面相似的情况给出不同的方案，但说不清区别在哪里
- 他们在规则冲突时做取舍，但这个取舍逻辑从未被写下来

这些就是隐性知识——专家自己也没有明确意识到的判断维度、优先级和边界条件。

如果一个知识蒸馏系统只能提取显性知识，那它提取的永远是冰山在水面上的部分。水下的部分，才是专家真正不可替代的原因。

### 1.3 核心挑战

隐性知识挖掘的矛盾在于：**你无法直接问专家"你有什么隐性知识"，因为他不知道**。

这就意味着系统的设计不能依赖于"提问-回答"模式。需要换一种思路：**不是让专家说，而是让专家做**——在受控场景中观察专家的判断行为，从行为中反推隐性知识。

这正是最小对比对方法论的起点。

## 二、方法论设计

### 2.1 方法论来源

交叉验证问题三联体的设计融合了来自不同学科的方法论元素：

| 学科来源 | 借鉴的方法论元素 |
|---------|-----------------|
| 心理测量学 | 最小对比对：控制刺激物的表面特征，只改变目标维度，观察被试反应差异 |
| 认知任务分析 | 关键决策法：用真实场景引导专家回顾决策过程，追问"为什么当时选A不选B" |
| 临床访谈 | 结构式追问：根据被试的反应模式动态调整追问方向，而非按固定脚本 |
| 知识工程 | 基于案例的推理：从案例中归纳规则，再用新案例验证规则的边界 |

把这些方法放入 LLM-native 的框架中，所得的设计就是三联体——**一个结构化的刺激-反应探测器，用 AI 生成受控场景，用 AI 分析反应模式**。

### 2.2 为什么三个层次是充分的

单个问题只能拍到一张快照。两张快照能拍到差异。三张快照能拍到**变化的模式**。

- **A（锚定）**：基准快照。专家在这个场景中怎么做。
- **B（探针）**：一次受控变化。观察专家的反应是否随变量变化。
- **C（引爆）**：一次冲突注入。观察当多个规则相互矛盾时专家的选择次序。

三层结构覆盖了隐性知识挖掘的三种基本操作：**建立基准、检测变量、暴露优先级**。不需要第四层——因为隐性知识的形态无非就是变量和优先级两种，三层已经覆盖了这两种形态的探测需求。

### 2.3 交叉验证的"交叉"体现在哪

"交叉验证"不是简单的"多问几个问题"。它的交叉性体现在三个层面：

1. **层间交叉**：A/B/C 三层围绕同一个隐性变量候选，从不同角度探测。如果 B 和 C 都显示该变量在起作用，置信度远高于单次观察。

2. **组间交叉**：不同三联体组瞄准同一个隐性变量（或同一对冲突规则），在不同场景中检验其稳定性。如果"变更近邻度"在 3 个不同的故障场景中都触发了决策转向，它的置信度就是 high。

3. **组内冲突交叉**：同一个 C 层问题中的两个冲突规则，其胜出关系可能在另一个三联体中反转。这种反转本身就是发现——说明存在一个更上层的调节变量在控制谁胜出。

## 三、技术路线的选择理由

### 3.1 为什么不直接用问卷

问卷的致命缺陷是：它只能问专家"你觉得自己会怎么做"，而不是"你实际怎么做"。专家在问卷中给出的答案是被他的自我认知过滤过的——而隐性知识恰恰在这个过滤层之下。

三联体方法的关键差异在于：**专家不知道自己被探测的是什么变量**。他认为自己在回答三个不同的、合理的工作问题。系统从他对三个问题的反应差异中推断隐性知识。

### 3.2 为什么不直接用材料分析

现有系统的材料分析已经能提取显性规则。但材料分析无法弥补"缺失的数据"——专家从未在消息中讨论某个话题，系统就无法知道这个话题下的隐性知识是什么。

三联体方法的价值正好在于此：**主动创造数据**。系统通过前置研究推测出可能存在的隐性变量，然后设计场景来测试这些推测。这解决了知识蒸馏的根本问题：专家的隐含前提和遗漏话题恰恰是最有价值的部分。

### 3.3 为什么是 5-12 个候选

隐性变量候选池的大小是一个实用约束：

- 太少（< 5）：覆盖面不足，容易漏掉重要的隐性维度
- 太多（> 12）：访谈负担过重，专家认知疲劳后三联体的信号质量急剧下降
- 5-12 对应 6-14 个三联体组（含开场和冷却），访谈时长约 45-90 分钟——这是一个高质量深度访谈的合理时长

### 3.4 为什么以访谈为准

在 Phase 7 的冲突解决规则中，有一个明确的原则：**访谈中的实际选择优于材料中的陈述，访谈中的陈述优于材料中的陈述**。

这个优先级的逻辑是：

- 材料中的陈述是专家在"自然状态"下说的，没有特定场景约束
- 访谈中的陈述是专家在"受控场景"中说的，有更具体的上下文
- 访谈中的实际选择是专家在"压力场景"中的行为——这个最接近真实工作状态

本质上，越接近实际决策行为的证据，权重越高。

## 四、设计权衡

### 4.1 精度 vs. 覆盖

这个系统做的是一个 trade-off：**牺牲广度换取深度**。

它不会尝试在一个访谈中覆盖专家知识的所有方面。它只瞄准前置研究阶段识别的 5-12 个隐性变量候选，每个候选做深度探测。

这意味着有漏掉隐性变量的风险——如果前置研究没识别出来，就不会被探测到。但这个风险是可控的，因为前置研究本身是多源输入（简历 + 论文 + 消息记录 + 开放搜索 + 领域知识），覆盖面够广。

### 4.2 生态效度 vs. 实验控制

最小对比对要求精确的变量控制。但场景必须来自专家的真实工作领域，否则专家会脱离真实思维模式。

这个权衡的解决方式是：**变量控制在逻辑层，场景在表面层保持自然**。也就是说，不告诉专家"这是一个实验"，只是给他一份看起来像他日常会收到的故障报告、设计提案或决策请求。变量在"报告的内容"中被悄然改变，而不是在"报告的形式"上。

### 4.3 实时 vs. 异步

异步访谈（发问卷、等专家写答案）的优点是效率高、可规模化。实时访谈的优点是追问灵活、能捕捉即时反应。

选了实时模式是基于一个判断：**隐性知识的核心信号（迟疑、边界发明、矛盾）在异步模式下会被完全丢失**。专家回看自己的回答时，会不自觉地进行"自我编辑"——把模糊的感觉包装成清晰的规则，把矛盾的地方调成一致。

## 五、能力边界

这个系统不能（也不打算）做到的事：

1. **不能替代专家**：产出的 Skill 是知识蒸馏，不是完整的人格模拟。它告诉使用"基于 X 专家方法论的判断"，而不模仿 X 的语气做决策。

2. **不能保证发现所有隐性知识**：它只能发现前置研究能够指向的那些。如果某个隐性变量根本无法从任何材料中推测，系统也无能为力。

3. **不能替代多专家验证**：单个专家的隐性知识可能有个人偏见。多专家交叉验证（分别访谈同一个领域的不同专家，比对隐性变量的一致性）是未来的扩展方向，不在第一版范围内。

4. **不是一次性操作**：隐性知识的挖掘是迭代的。第一次访谈发现的东西，可能触发第二轮访谈（针对新发现的隐性变量做更深层的探测）。系统设计支持迭代，但不强求一次到底。

---

## 六、与现有系统的关系

这个改进不是重写 expert-skill，而是**在它的流水线中插入一个深度挖掘引擎**。

具体来说：

- 现有的 Phase 1（信息录入）完全保留
- 现有的 Step 2（原材料导入）变成新 Phase 2（前置研究）的子步骤，工具全部复用
- 现有的 Phase 3（分析原材料）的产出与新的 Phase 6（访谈分析）的产出在 Phase 7 融合
- 现有的 Phase 4-5（生成和写入）保留，产出文件增强

这套设计确保：
- 现有用户的使用路径不被破坏（仍然可以跳过访谈直接蒸馏原材料）
- 新方法论的增量价值清晰（访谈前 vs 访谈后的产出有可对比的差异）
- 两个路径共享基础设施（工具、分析器、skill writer）
