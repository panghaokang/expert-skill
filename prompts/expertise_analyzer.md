# 专家知识分析 Prompt

## 任务

你将收到 **{name}** 的原材料（文档、消息、邮件等），以及他的专长类型 **{expertise_type}**。

从中提取他的专业知识和判断框架，用于构建 Expert Skill。

**原则：只提取专业相关内容。有依据才写，没有就标注"原材料不足"。不做性格推断，不分析沟通风格。**

---

## 通用提取维度

### 1. 专业领域

- 他负责的系统/模块/业务
- 他的核心技术栈与工具链
- 他的职责边界（哪些是他的领域，哪些不是）

```
输出格式：
领域：[描述]
核心系统：[列表]
技术栈：[列表]
边界：[管什么/不管什么]
```

### 2. 判断框架与启发式

- 他遇到问题时优先检查什么
- 他的分析方法（先看什么、后看什么、为什么是这个顺序）
- 他的经验规则（"遇到 X 就查 Y"这类模式）
- 直接引用原话中的判断逻辑

```
输出格式：
优先检查项：[按优先级排列]
分析路径：[步骤描述]
经验规则：
  - "[原话或提炼]"
  - "[原话或提炼]"
```

### 3. 决策模式

- 做技术决策时的优先考量（正确性/性能/可维护性/速度/成本）
- 什么情况下他会说不/反对
- 他的技术红线

```
输出格式：
决策考量：[排序]
技术红线：[列表]
反对模式：[他何时说"不行"以及为什么]
```

### 4. 知识结构

根据专长类型，提取对应结构的知识：

**诊断专家 (troubleshooter)**：
- 故障模式库：他经常处理的故障类型
- 诊断路径：从症状到根因的排查链
- 嗅觉检查：一上来就查的固定检查项

**架构专家 (architect)**：
- 设计原则：他的架构哲学与优化方向
- 技术选型矩阵：什么场景选什么方案
- 反模式：他认为"绝对不要这样设计"的做法

**审核专家 (reviewer)**：
- 缺陷模式：他反复发现的典型问题
- 检查清单：Review 时的固定检查项
- 风险等级判断：什么算 blocker，什么算 nit

**决策专家 (decision_maker)**：
- 决策框架：他做决策的方法论
- 风险评估：他如何评估风险和不确定性
- 优先级方法：如何排序和取舍

**运维专家 (operator)**：
- 监控指标：他关注的指标和阈值
- SOP：标准操作流程
- 升级规则：何时升级、升级给谁

### 5. 经验知识库

他明确表达的经验判断、踩过的坑、技术观点（直接引用原话）：

```
- "[原话或结论]"
- "[原话或结论]"
```

---

## 输出要求

- 语言：中文
- 没有信息的维度：标注 `（原材料不足，建议追加相关文档）`
- 有原文依据的结论：加引号标注原话
- 输出结果要求具体可执行，不要写"可能""倾向于"这类模糊表述
- 不要分析他的性格、沟通风格、或对人对事的态度
